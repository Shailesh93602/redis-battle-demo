How it works
Pattern 1
@socket.io/redis-adapter
Without an adapter, Socket.io rooms only work within a single process. Two instances can't emit to the same room.

The Redis adapter uses pub/sub to relay events across instances. Requires two separate clients: one for publish, one for subscribe — a subscribed connection cannot send commands.

io.adapter(createAdapter(pubClient, subClient))
Pattern 2
Redlock (distributed mutex)
Every 2 seconds all running instances race to acquire battle:tick:lock. One wins; the others skip. Without this, every instance would emit a server_tick — clients would receive N duplicates per interval.

retryCount: 0 is intentional. A delayed tick is worse than a skipped tick in a real-time game context. The loser skips immediately rather than queuing.

The tick bar above shows which instance won each race.
Why it matters
Horizontal scaling
This is the exact architecture behind EduScale's real-time coding battles. With a single Node.js instance, neither pattern is needed. But any load balancer or auto-scaling group creates these problems.

Open this demo in two tabs pointing to different ports (:3001 and :3002, both backed by the same Redis) to watch the lock flip between instances.
